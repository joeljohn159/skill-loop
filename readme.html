Claude Code plugin

skill-loop

Learn a repository's conventions once, then keep them current automatically — staying near-zero tokens on the sessions where nothing notable happened.

Version 0.1.0MIT licensedBash & Markdown, no runtime deps

01 Overview

The first run reads the codebase and generates exactly the skills Claude needs so it won't violate the project's conventions on the next pull request. Every session afterward, the plugin refines those skills on its own.

It is organised as six layers. The expensive, high-capability work is rare and explicit; the constant work is cheap or free.

LayerTriggerCostResponsibility
BootstrapLayer 1commandOpus, once Crawl the repo. Judgment rules become small skills; mechanically enforceable rules become formatter and linter configuration. Every rule ships with a command that verifies it.
EnforceLayer 2PostToolUse0 tokens Run the project formatter on each edited file. Deterministic rules are auto-fixed and never become skills.
CaptureLayer 3hooks0 tokens Append raw signals to a queue — corrections, new patterns, approvals, and failures.
ReflectLayer 4Stop hookgated A deterministic pre-scan first. With no signal it exits free; otherwise a single Haiku pass reads only the flagged diffs, extracts and de-duplicates candidate rules, and stages them with a recurrence count.
PromoteLayer 5commandSonnet, rare Candidates that have recurred are generalised, written into the correct skill, pruned of stale entries, and committed to git one change at a time.
CI feedbackLayer 6commandmodel_ci Paste a failing build log from any CI; it and your local fix become a high-value failure candidate. Runs locally — nothing is installed in the CI runner.

02 Installation

Add the marketplace and install the plugin:

/plugin marketplace add joeljohn159/skill-loop
/plugin install skill-loop@skill-loop

Then run the one-time bootstrap inside the repository you want it to learn:

/skill-loop:bootstrap

After that, work as usual. The hooks take over and the skills evolve in the background.

Watching it work

Run /skill-loop:logs to open a new terminal tab beside the session, streaming a clean, colorized account of what the plugin is doing — separate from the verbose debug log.

13:05:58  SKILL        used: sl-error-handling          ← a learned skill fired
13:06:00  CORRECTION   a.ts: user edited Claude's output after the write
13:06:00  REFLECT      reflecting on 2 signal(s) · haiku
13:06:00  REFLECT      staged 1 candidate(s); 0 ready to promote
13:06:00  PROMOTE      sl-error-handling ← Return Result for expected errors

On iTerm it opens a tab; on Terminal a tab or window. Anywhere else — a VS Code terminal or a remote box — it prints the single command to run yourself.

03 What it learns from

Capture is pure heuristics — four kinds of signal, so it learns from anything new, not only mistakes.

Correction
You edit a file Claude wrote. Detected by hashing the write ledger against disk, with a real before-and-after diff preserved.
New pattern
You introduce a library or tool Claude didn't use — install commands and dependency-manifest changes.
Approval
A task completes cleanly: a commit, merge, or push that finished without error.
Failure
A command, test, or build errors — including continuous-integration failures via Layer 6.

Learning runs locally on the Claude you're already using. /skill-loop:learn reflects on demand over the current chat (optionally with an explicit lesson); /skill-loop:learn-from-ci takes a pasted red-build log from any provider. Your CI never runs Claude.

04 Token model

  • The Stop reflection is the only per-session model call, and it is skipped entirely when the pre-scan finds no signal.
  • Model tiering is strict: Haiku for capture, extraction, and de-duplication; Sonnet or Opus only for bootstrap and promote.
  • Skills stay small and rely on progressive disclosure — the description loads, the body loads on a relevance match. Session start injects only a compact index, never the whole corpus.
  • De-duplication and a recurrence threshold gate every write, so skill files never bloat. Bloat is paid for in tokens on every future session.

Choosing your models

The tiers above are the Balanced default. Run /skill-loop:configure at any time — or answer the prompt bootstrap shows on first run — to change them. A user who isn't worried about token cost can put Opus on every stage.

ProfileBootstrapPromoteReflectCI
Maximumopusopusopusopus
Balancedopussonnethaikuhaiku
Economysonnethaikuhaikuhaiku
Customchoose each stage individually

reflect is the only automatic per-session call, so its model is the main cost lever. The choice is stored in .skill-loop/config and applies immediately. For headless or CI use, pass the profile as an argument — no prompt.

06 Safety

  • Automatic learning writes only to staging. A person runs promote, which commits each change to git — so every step in a skill's evolution is reviewable and revertable.
  • Concurrent sessions reflecting into the same files use an atomic directory lock and merge by recurrence count rather than overwrite.
  • Hooks are defensive. Each exits cleanly and degrades gracefully when an optional tool or a well-formed input is missing, so a session is never blocked.
